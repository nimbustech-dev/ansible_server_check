# 배포 자동화 업그레이드 옵션

## 현재 상태
- 수동 배포: `./deploy_current_branch.sh` 실행 필요
- GitHub 저장소 사용 중
- 서버: 네이버 클라우드 (27.96.129.114:4433)

## 자동화 옵션

### 옵션 1: GitHub Actions (CI/CD) - 추천 ⭐

**개념**: GitHub에 push하면 자동으로 배포

**장점**:
- ✅ GitHub에서 직접 관리
- ✅ 배포 히스토리 확인 가능
- ✅ 여러 브랜치별 배포 전략 가능
- ✅ 배포 전 테스트 실행 가능
- ✅ 롤백 자동화 가능

**단점**:
- ⚠️ GitHub Secrets에 SSH 키 등록 필요
- ⚠️ GitHub Actions 사용량 제한 (무료 플랜)

**구현 방법**:
1. `.github/workflows/deploy.yml` 생성
2. `develop` 브랜치 push 시 자동 배포
3. SSH 키를 GitHub Secrets에 등록

**예상 시간**: 30분

---

### 옵션 2: Git Hooks (로컬 자동화)

**개념**: 로컬에서 `git push` 시 자동 배포

**장점**:
- ✅ 로컬에서 즉시 배포
- ✅ 추가 서비스 불필요
- ✅ 빠른 반응

**단점**:
- ⚠️ 로컬에서만 작동 (다른 개발자는 수동)
- ⚠️ 네트워크 연결 필요

**구현 방법**:
1. `.git/hooks/pre-push` 생성
2. `develop` 브랜치 push 시 `deploy_current_branch.sh` 실행

**예상 시간**: 10분

---

### 옵션 3: 서버 측 Webhook (GitHub Webhook)

**개념**: GitHub에서 push 이벤트 발생 시 서버가 자동 pull

**장점**:
- ✅ 서버에서 자동으로 최신 코드 pull
- ✅ GitHub과 직접 연동
- ✅ 여러 개발자 동시 작업 가능

**단점**:
- ⚠️ 서버에 webhook 리스너 필요
- ⚠️ GitHub Webhook 설정 필요
- ⚠️ 보안 설정 필요 (secret 검증)

**구현 방법**:
1. 서버에 webhook 리스너 스크립트 생성
2. GitHub Webhook 설정
3. push 이벤트 수신 시 `auto_update_smart.sh` 실행

**예상 시간**: 1시간

---

### 옵션 4: 스케줄링된 자동 배포 (Cron)

**개념**: 주기적으로 변경사항 확인 후 배포

**장점**:
- ✅ 간단한 구현
- ✅ 서버 리소스 적게 사용

**단점**:
- ⚠️ 실시간 배포 아님 (지연 발생)
- ⚠️ 불필요한 배포 시도 가능

**구현 방법**:
1. 서버에 cron job 추가
2. 주기적으로 `git fetch` 후 변경 감지
3. 변경 있으면 배포 스크립트 실행

**예상 시간**: 20분

---

### 옵션 5: 하이브리드 (GitHub Actions + 서버 Webhook)

**개념**: GitHub Actions로 배포 + 서버에서 자동 pull

**장점**:
- ✅ 이중 안전장치
- ✅ 배포 히스토리 + 자동 동기화

**단점**:
- ⚠️ 복잡도 증가

---

## 추천 순서

### 1순위: GitHub Actions (옵션 1)
- 가장 표준적이고 안정적
- 배포 히스토리 관리 용이
- 여러 브랜치 전략 가능

### 2순위: Git Hooks (옵션 2)
- 빠른 구현
- 로컬 개발 환경에 적합

### 3순위: 서버 Webhook (옵션 3)
- 실시간 자동화
- 여러 개발자 협업에 적합
