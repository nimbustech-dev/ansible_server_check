# 로그인·회원가입 강화 준비 가이드

현재 구조를 기준으로, **어떤 걸 강화할지**와 **구현 순서**를 정리한 문서입니다.

---

## 1. 현재 구현 요약

| 구분 | 내용 |
|------|------|
| 로그인 | 아이디/비밀번호, JWT(쿠키), 승인된 사용자만 로그인 |
| 회원가입 | 아이디, 비밀번호, 이메일(선택), 관리자 승인 후 로그인 |
| 비밀번호 | bcrypt 해시 저장, 72바이트 제한 |
| 인증 | JWT HS256, 24시간, `session` 쿠키 (HttpOnly, SameSite=Lax) |

---

## 2. 강화할 수 있는 영역

### A. 입력 검증 (우선 적용 권장)

- **아이디**
  - 길이: 예) 4~20자
  - 허용 문자: 영문, 숫자, `_` 등 (한글/특수문자 정책 결정)
- **비밀번호**
  - 최소 길이: 예) 8자 이상
  - 복잡도: 대문자, 소문자, 숫자, 특수문자 중 N종 포함 (선택)
  - 회원가입 시 **비밀번호 확인** 필드 추가
- **이메일**
  - 형식 검증 (프론트 + 백엔드)
  - 선택이면 “미입력” 허용, 입력 시에만 검증

**준비할 것**

- 프론트: `login_template.html`, `register_template.html`에
  - 비밀번호 확인 필드(회원가입)
  - `minlength` / `maxlength` / `pattern` 등 HTML5 검증
  - 필요 시 JS로 실시간 메시지
- 백엔드: `LoginRequest`, `RegisterRequest`에 Pydantic `Field`/`validator`로 동일 규칙 적용 (중복 가입, 길이, 패턴 등)

---

### B. 보안

- **로그인/회원가입 제한**
  - IP 또는 계정당 시도 횟수 제한 (예: 5회 실패 시 15분 잠금)
  - 라이브러리: `slowapi`(FastAPI 연동) 또는 Redis/메모리로 직접 구현
- **비밀번호 정책**
  - 최소 길이 + 복잡도 규칙을 백엔드에서 검사 (프론트만 믿지 않기)
- **JWT**
  - 운영 환경에서는 `JWT_SECRET_KEY` 반드시 환경변수로 설정
  - 필요 시 액세스 토큰 짧게 + 리프레시 토큰 도입 검토

**준비할 것**

- `requirements.txt`에 rate limit용 패키지 추가 (예: `slowapi`)
- 실패 횟수 저장소 결정: 메모리 dict vs Redis (서버 재시작 시 초기화 여부 결정)
- `.env.example`에 `JWT_SECRET_KEY` 등 보안 관련 변수 설명 추가

---

### C. UX·기능

- **비밀번호 보기/숨기기** 토글 (로그인·회원가입 공통)
- **회원가입**
  - 비밀번호 확인 입력 + “일치하지 않습니다” 메시지
  - 아이디/이메일 중복 여부 실시간 확인 API (선택)
- **에러 메시지**
  - “아이디 또는 비밀번호가 올바르지 않습니다” 유지 (계정 존재 여부 노출 X)
  - 회원가입 실패 시 필드별 메시지 (예: “비밀번호가 약합니다”)

**준비할 것**

- 공통 CSS/JS 스니펫 (비밀번호 토글, 메시지 스타일)
- 중복 확인이 필요하면 `/api/auth/check-username`, `/api/auth/check-email` 같은 API 설계

---

### D. 선택 기능 (나중에 확장)

- 이메일 인증: 가입 시 인증 링크 발송
- 비밀번호 찾기: 이메일로 재설정 링크
- 리프레시 토큰: 액세스 토큰 만료 시 갱신
- 2FA(OTP)

이 경우 메일 발송 설정(SMTP), 토큰 저장소(DB/Redis), 만료 정책을 미리 설계해 두면 좋습니다.

---

## 3. 구현 순서 제안

1. **1단계 (바로 적용 가능)**  
   - 회원가입: 비밀번호 확인 필드 + 프론트 검증  
   - 백엔드: `RegisterRequest`/`LoginRequest`에 길이·패턴 검증 (Pydantic)  
   - 비밀번호 보기/숨기기 토글  

2. **2단계**  
   - 로그인/회원가입 rate limit (같은 IP 또는 계정)  
   - 비밀번호 복잡도 규칙 (백엔드)  

3. **3단계**  
   - 아이디/이메일 실시간 중복 확인 (필요 시)  
   - 이메일 인증·비밀번호 찾기 등 선택 기능  

---

## 4. 수정 대상 파일 정리

| 목적 | 파일 |
|------|------|
| 회원가입 폼·검증 | `register_template.html` |
| 로그인 폼·검증 | `login_template.html` |
| 요청 모델·검증 | `main.py` (LoginRequest, RegisterRequest) |
| 비밀번호 정책 | `database.py` 또는 `main.py` (가입 시 검사) |
| Rate limit | `main.py` (미들웨어/의존성), 새 모듈 가능 |
| 환경 변수 예시 | `.env.example` |

이 순서대로 적용하면 로그인·회원가입을 단계적으로 강화할 수 있습니다.  
원하면 1단계부터 구체적인 코드 패치 예시도 작성해 줄 수 있습니다.
